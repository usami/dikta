# Dikta — Design Document

## What is Dikta?

Dikta is a TypeScript library that replaces traditional ORMs for the age of AI coding agents.

The name comes from Latin *dictare* (to dictate, to declare). You dictate your intent and constraints; agents execute.

## The Problem

ORMs were designed to solve two problems for human developers:

1. **Productivity** — Humans find it tedious and error-prone to write SQL by hand for every operation.
2. **Impedance mismatch** — Object-oriented type systems and relational models don't map cleanly to each other.

When AI coding agents become the primary authors of application code, problem #1 disappears entirely. Agents can generate correct SQL at near-zero cost. The productivity abstraction that ORMs provide becomes overhead rather than value.

Problem #2 remains, but its solution changes shape. Instead of a runtime abstraction layer (ORM) that translates between objects and tables at execution time, what's needed is a **declarative contract system** that tells agents *what to generate* and *how to verify it*.

## Why Not Prisma / Drizzle / sqlc?

These tools each moved in the right direction but stopped short:

- **Prisma** made schemas declarative and generates typed clients, but the schema captures only structure — not intent, policy, or performance expectations. There is no way to say "this field is PII" or "this query must use an index scan."
- **Drizzle** is SQL-first and type-safe, but it's still a runtime query builder designed for humans to compose queries interactively. Agents don't need a fluent API.
- **sqlc** generates code from SQL, which is the right direction, but it requires hand-written SQL as input and has no mechanism for declaring constraints, policies, or verification.

Dikta takes the next step: the schema and query definitions carry **semantic metadata** — intent, policy, invariants, performance contracts — that agents use as guardrails when generating code, and that automated verification uses to ensure correctness.

## Design Principles

### 1. Intent over Abstraction

Traditional ORMs abstract away SQL behind method chains and model classes. Dikta does not abstract — it **annotates**. Each field, entity, and query carries metadata about *why* it exists, *what rules govern it*, and *how it should behave*. This metadata is meaningless to a runtime query builder, but invaluable to an agent deciding how to generate code.

```
ORM:        User.findMany({ where: { active: true } })
            → "what" is clear, "why" and "how safely" is implicit

Dikta:      defineQuery("getActiveUsers", {
              purpose: "Dashboard user list",
              security: { requires: ["tenant_isolation"] },
              performance: { scan_strategy: "index_only" },
            })
            → "what", "why", "how safely" are all explicit
```

### 2. Contract over Convention

ORMs rely on conventions (naming, directory structure, lifecycle hooks) that human developers learn and internalize over time. Agents work better with explicit, machine-readable contracts. Dikta's Query Contracts declare exactly what a query must do — its parameters, return shape, performance bounds, security requirements — so that any implementation (whether generated by an agent or written by a human) can be verified against the contract.

### 3. Generated over Maintained

ORMs are libraries: you install them, import them, call their API, and maintain compatibility with their versions over time. Dikta's access layer is **generated code** — disposable output files that can be regenerated whenever the schema or contracts change. There is no runtime dependency to keep up to date. The only things you maintain are the schema definitions and contracts; everything else is ephemeral.

### 4. Verification over Trust

When a human writes ORM code, code review is the primary quality gate. When an agent generates SQL, you need automated verification. Dikta's contracts serve as test specifications: generated SQL can be checked for LIMIT clauses, security filters, index usage, and other contract requirements — without running the queries against a database.

## Architecture Overview

```
Human / upstream agent maintains:
  ┌─────────────────────────────────────────────┐
  │  Intent Schema    — entities, fields, roles, │
  │                     policies, invariants     │
  │  Query Contracts  — params, return shape,    │
  │                     performance, security    │
  │  Policies         — global rules, tenant     │
  │                     isolation, PII handling  │
  └─────────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────┐
  │  Dikta Generator                             │
  │  (or: an agent reading the above contracts)  │
  └─────────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────┐
  │  Generated artifacts (.generated/)           │
  │  - DDL (CREATE TABLE, INDEX)                 │
  │  - Typed access functions                    │
  │  - Validators (invariant checks)             │
  │  - Contract verification tests               │
  │  - Migration SQL                             │
  └─────────────────────────────────────────────┘
                        │
                        ▼
  ┌─────────────────────────────────────────────┐
  │  Verification Layer                          │
  │  - Type checking (tsc)                       │
  │  - EXPLAIN analysis                          │
  │  - Invariant tests                           │
  │  - Policy compliance checks                  │
  └─────────────────────────────────────────────┘
```

The key insight: **humans manage intent (top box), machines manage implementation (bottom boxes)**. The middle layer — generation — can be Dikta's built-in generator OR an external agent that reads the same contract files. Dikta doesn't monopolize code generation; it provides the contract format and verification tools.

## Core Concepts

### Intent Schema

A schema definition that carries more than structure. Each field has a **role** (identifier, monetary, audit timestamp, etc.), **policies** (PII handling, retention, exposure rules), and entities have **invariants** (business rules that must always hold).

### Query Contract

A declaration of what an application needs from the database. Not a query builder — a specification. It states: "I need data in this shape, with these parameters, meeting these performance and security constraints." Any implementation that satisfies the contract is correct.

### Ephemeral Access Layer

The generated TypeScript functions that actually call the database. These are `.generated/` files, gitignored, regenerated on demand. They have no API stability guarantees. They exist only to satisfy the contracts.

### Agent Context

A machine-readable summary of the entire project's data layer — entities, relationships, PII fields, performance budgets, policies — that an agent reads first to understand the constraints before generating any code.

## Technology Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Language | TypeScript 5.x, strict | Target audience is TS ecosystem |
| Package structure | pnpm monorepo | Separate core, generator, migration packages |
| Build | tsup | Fast, simple, handles DTS generation |
| Test | vitest | Fast, native TS support, good DX |
| Validation | zod | Already standard in TS ecosystem, composable |
| CLI | citty or commander | Lightweight, well-maintained |
| First DB target | PostgreSQL | Most common for serious applications |
| SQL parser | pgsql-ast-parser (or minimal) | For contract verification against SQL |

## Naming Conventions

| Concept | Name | Origin |
|---|---|---|
| Schema definition | Intent Schema | Schema + intent metadata |
| Query specification | Query Contract | Formal contract, not a builder |
| Field-level rules | Field Policy | Per-field data governance |
| Business rules | Invariant | From formal methods |
| Generated output | Artifact | Disposable, regeneratable |
| Agent summary file | Agent Context | Context for AI agents |
| Contract failure | Violation | Clear severity signal |

## Development Phases

| Phase | Scope | Depends on | Completion criteria |
|---|---|---|---|
| 1 | Intent Schema engine | — | `defineEntity` with type inference, serializable |
| 2 | Query Contract system | Phase 1 | `defineQuery` with type inference, schema validation |
| 3 | Code generation | Phase 1 + 2 | CLI generates DDL + access functions + validators |
| 4 | Migration planner | Phase 3 | Schema diff → safe migration SQL |
| 5 | Agent protocol | Phase 1+ | agent-context.json + violation reporter |

Phase 1 is the foundation. Phase 3 is the most work. Phase 5 can start early and evolve continuously.

## Non-Goals (for now)

- **Runtime query builder** — Dikta is not an ORM. No `.where().join().select()`.
- **Multi-database support** — PostgreSQL first. MySQL target in progress via `SQLDialect` abstraction layer (see ROADMAP.md).
- **GUI / dashboard** — CLI and file-based workflows only.
- **Schema migration execution** — Dikta generates migration SQL but does not run it. Use existing tools (dbmate, flyway, etc.) for execution.
